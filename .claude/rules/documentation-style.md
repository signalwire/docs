# Documentation Style & Structure

This rule defines **how every documentation page under `fern/` is structured, styled, and formatted**, so all pages stay 1:1 consistent on the Fern docs site. It applies whenever you author or substantially edit an `.mdx` page.

> **Companion rule вҖ” `docs-conventions.md`:** covers *which* docs are hand-authored vs auto-generated (e.g. REST API reference is generated from TypeSpec вҖ” never hand-edit it) and how to apply audit findings. This rule covers *how* to write a page; read both together.

The rules here are **product-agnostic** вҖ” they apply to SWML, the server SDKs, and any other product. Examples use **FreeSWITCH** (modules, dialplan applications, CLI commands, channel variables, events) as a concrete running example, but the patterns are universal.

## 1. How to use this rule

You are authoring documentation pages in **MDX v3** for the SignalWire docs site, built on the **Fern** platform. Your input is some **source of truth to document** вҖ” source code, an SDK interface, a markup or config spec, an API definition, or an event/callback contract вҖ” and your job is to turn it into pages that match the SignalWire house style exactly.

**Your contract вҖ” every page you emit must:**

1. Be a valid **MDX v3** file (see [В§7](#7-strict-mdx-v3-rules) вҖ” this is non-negotiable; loose HTML breaks the build).
2. Begin with complete, correct **front matter** ([В§4](#4-front-matter)).
3. Match the **structure for its page type** ([В§5](#5-page-types-and-skeletons)).
4. Use only components from the **component catalog** ([В§6](#6-component-catalog)).
5. Document **only what exists in the source** вҖ” never invent parameters, return values, defaults, or behavior.

**Per-page workflow вҖ” follow this every time:**

```
1. READ the source. Identify exactly: name, parameters (name/type/required/default),
   return value, observable behavior, and a realistic usage example.
2. CLASSIFY the page type (reference / overview / guide / landing) вҖ” В§5.
3. PLACE the file in the folder tree so the nav generates correctly вҖ” В§3.
4. FILL the matching skeleton вҖ” В§5. Copy only facts from the source.
5. SELF-CHECK against the checklist in В§9 before emitting. Fix every miss.
6. EMIT the file. Nothing else вҖ” no commentary outside the file unless asked.
```

If the source does not tell you something (e.g. a default value), **omit it** rather than guessing. If a whole section has no source basis (e.g. there are no parameters), **drop the section** вҖ” do not write "None" or fabricate one.

## 2. Core principles

- **Parity over creativity.** The site already has a house style. Match it exactly; do not introduce new patterns, components, or section names.
- **MDX v3 strict.** The build uses MDX 3 with Acorn JSX parsing. HTML-era habits (`<!-- -->`, unclosed `<img>`, bare `<` in prose) are hard build errors, not warnings.
- **The folder tree *is* the navigation.** Nav is auto-generated by pointing Fern at a folder. Organize files exactly as you want the sidebar to read ([В§3](#3-file-and-folder-structure)).
- **Don't invent.** Source code is the single source of truth. No parameter, type, default, or behavior appears in the docs unless it appears in the source.
- **One concept per file.** Mirrors the existing reference docs and keeps the nav clean.

## 3. File and folder structure

The SignalWire docs auto-generate their sidebar from the folder layout вҖ” a Fern nav entry like the one below points at a directory and builds the section from the files inside it, taking each item's label from its front-matter `title`:

```yaml
# How the live site wires a folder into the nav (for context вҖ” you do NOT write this file):
- folder: ./pages/reference/methods/calling
  title-source: frontmatter      # sidebar labels come from each page's `title:`
  title: Calling                 # the section heading
```

**What this means for you:** you don't write nav config. You produce a clean folder tree, and the nav follows. So the tree must read like the sidebar you want.

**Folder path and slug are independent.** The folder layout drives the **navigation**; the
front-matter `slug` drives the **URL** ([В§4](#slug-standards)). Keep them separate вҖ” a page's
slug should *not* echo its filing folders вҖ” so you can reorganize sections later without
changing any URLs.

### Rules

- **Split `reference/` from `guides/`.** Reference = exhaustive, lookup-oriented (one page per module/command/variable). Guides = task-oriented, narrative (how to accomplish something).
- **One concept per file.** `kebab-case.mdx`. Examples: `originate.mdx`, `channel-variables.mdx`, `mod-dptools.mdx`.
- **Simple concept вҶ’ a single file.** `<name>.mdx`.
- **An area or concept with its own pages вҶ’ a folder (= a sidebar section).** Whether that folder gets an `overview.mdx`, an `index.mdx`, or no landing page at all depends on the section type вҖ” see [Section landing pages](#section-landing-pages) directly below. This is the most important structural decision, so get it right.
- **Ordering is alphabetical by default вҖ” leave it that way.** Don't set `position` on normal pages. The one exception: a section's **`overview.mdx` sets `position: 0`** so it leads instead of sorting into the middle. Only add `position` elsewhere when a deliberate non-alphabetical order is genuinely needed.

### Section landing pages

A folder renders as a sidebar section. There are **three** ways to set up a section, depending on whether it needs a landing page and whether its header should be clickable:

| Section type | Landing file | Sidebar behavior | Use for |
|--------------|--------------|------------------|---------|
| **Top-level section** | `overview.mdx` (with `position: 0`) | The section label is a **non-clickable** header, rendered flat (expanded); the landing appears as an "Overview" child page. `position: 0` keeps it at the top вҖ” without it, "Overview" sorts alphabetically into the list. | The main sections directly under a tab вҖ” a namespace or top-level area. |
| **Nested topic subsection** | `index.mdx` | The subsection label **is clickable** and links to the landing page; its detail pages nest beneath it. | A nested subsection that is itself a topic worth a landing page (a class, a complex multi-page method or module). |
| **Organizational subsection** | *(neither)* | The label is just a collapsible grouping вҖ” nothing to click through to; it only lists its child pages. | A nested grouping that exists only to organize/list pages and isn't a topic on its own. |

**The rule:** top-level вҶ’ `overview.mdx`; nested topic вҶ’ `index.mdx`; pure organization вҶ’ no landing file. The reason for the `overview.mdx`/`index.mdx` split: `overview.mdx` keeps the top-level header a non-clickable label (overview reachable as its first child), while `index.mdx` is what makes a nested subsection header click through to its landing page.

Real examples in the repo:
- **Top-level вҶ’ `overview.mdx`:** the server-SDK namespaces (`reference/python/agents/overview.mdx`, `.../core/overview.mdx`, `.../relay/overview.mdx`) and the SWML method categories (`reference/methods/calling/overview.mdx`).
- **Nested topic вҶ’ `index.mdx`:** a server-SDK class with its methods nested as siblings (`agents/agent-base/index.mdx`), the `relay/actions/index.mdx` group, and a complex SWML method with sub-pages (`calling/ai/index.mdx`).
- **Organizational, no landing:** the REST API reference groups endpoints under bare sections (e.g. a `Video` grouping) that exist only to organize, with no landing page.

## 4. Front matter

Every `.mdx` file starts with a YAML front-matter block delimited by `---`.

### Required on every page

**All four of these fields are required on every page вҖ” no exceptions.** A page missing any of
them is incomplete:

```yaml
---
title: "originate"                              # page title + sidebar label
slug: /reference/api-commands/originate          # stable URL вҖ” see Slug standards below
description: Originate a new call from FreeSWITCH to a destination.   # one sentence, for SEO/search
max-toc-depth: 3                                 # always exactly 3, on every page
---
```

| Field | Rule |
|-------|------|
| `title` | The page's **H1** вҖ” Fern renders it as the page heading and sidebar label (see the callout below). For reference pages, use the **exact symbol name** (e.g. `originate`, `channel-variables`); quote it if it contains characters YAML would choke on. |
| `slug` | The page's URL, **required on every page**. See [Slug standards](#slug-standards) below. |
| `description` | **Required on every page.** One sentence describing the page; used in search results and meta tags. Never leave it blank or omit it. |
| `max-toc-depth` | **Required on every page, always exactly `3`.** Set it verbatim вҖ” never omit it and never use another value. |

> **The H1 is the `title`, not body Markdown.** Every page has exactly one H1, and it comes
> from the `title` front-matter field. Do **not** write a `#` heading anywhere in the body вҖ”
> it would produce a duplicate, wrongly-styled heading. The body starts with the description
> paragraph, and all body section headings are `##` or deeper.

### Slug standards

**Every page must declare a `slug`.** The slug is the page's URL вҖ” and it is **independent of
where the file lives on disk**. This is deliberate: the folder tree drives the navigation
([В§3](#3-file-and-folder-structure)), the slug drives the URL, and keeping them separate means
you can reorganize folders (move a page into a different section) **without breaking its URL**.

Rules:

- **Decouple from the folder path вҖ” drop filing/grouping folders.** Name the concept and its
  stable domain, not the directory nesting. The live docs do exactly this:

  | File on disk | Slug |
  |--------------|------|
  | `reference/methods/calling/answer.mdx` | `/reference/calling/answer` (drops `methods/`) |
  | `guides/basics/deployment.mdx` | `/guides/deployment` (drops `basics/`) |
  | `guides/recipes/creating-ivr-with-swml.mdx` | `/guides/ivr` (drops `recipes/`, shortened) |

  `calling` survives because it's a stable product domain; `methods`, `basics`, `recipes` are
  filing buckets, so they're left out. If `deployment.mdx` later moves from `basics/` to
  `advanced/`, its slug stays `/guides/deployment`.

- **Keep them short and descriptive.** Two to three segments is the norm
  (`/reference/calling/answer`, `/guides/ivr`). The slug can be shorter and cleaner than the
  filename вҖ” favor the clearest concept name a reader would recognize.

- **Reference slugs may mirror a meaningful hierarchy.** Where the hierarchy *is* the stable
  structure (e.g. an SDK's `language/area/class/method`), mirror it:
  `/reference/python/agents/agent-base/on-debug-event`. Mirror meaning, never just folders.

- **Casing & spacing:** all **lowercase**, words separated by **hyphens** (kebab-case). No
  spaces, no camelCase, no underscores. Convert an underscore-named symbol to hyphens for the
  slug (`on_debug_event` вҶ’ `on-debug-event`). *Exception:* a reference page may keep a symbol's
  exact underscore name when matching the literal name aids recognition (e.g. an `ai_sidecar`
  method вҶ’ `/reference/calling/ai_sidecar`).

- **No product prefix, leading slash, no trailing slash, no extension.** Start at the
  product-relative root (`/reference/...`, `/guides/...`); the product config prepends
  `/docs/<product>` automatically. So the page slug is `/reference/calling/answer`, never
  `/docs/swml/reference/calling/answer` and never `reference/calling/answer.mdx`.

- **Choose a slug you won't have to change.** A stable slug avoids broken links and redirects.

### Optional (use when it applies)

| Field | When to use |
|-------|-------------|
| `subtitle` | A secondary line under the title. Common on overview/landing/guide pages, rare on reference pages. |
| `sidebar-title` | Override the sidebar label when it should differ from `title` (e.g. `Overview`). |
| `position` | Integer sort order within a folder (`0` = first). **Set `position: 0` on every `overview.mdx`** so it leads its section. Otherwise leave `position` off and let pages sort alphabetically вҖ” only add it elsewhere when a deliberate non-alphabetical order is truly needed. |
| `x-custom.tags` | Content tags, e.g. `product:voice`. Use only if you have a real taxonomy to apply. |
| `hide_title` | `false`/`true` to control the rendered H1. Leave off unless you have a reason. |
| `hidden` | `true` to keep a page out of the nav but still reachable. |

### Do NOT

- **Do not invent an `id` (UUID).** Some older pages carry one; new pages do not need it. Never fabricate a UUID.
- **Do not add a body `# H1`** вҖ” the `title` renders it (see the H1 callout above).

## 5. Page types and skeletons

Classify every page as one of four types and use its skeleton verbatim. Copy the skeleton, then fill it from the source.

**Heading conventions (all page types):**
- The H1 is the front-matter `title` вҖ” body headings are `##` or deeper, never `#` ([В§4](#4-front-matter)).
- **Use sentence case, not title case.** Capitalize only the first word and any proper nouns,
  acronyms, or code identifiers. Write `## Next steps`, `## From a web server`, `### Play a local file` вҖ”
  not `## Next Steps`, `## From A Web Server`, `### Play A Local File`. Acronyms and literal names keep
  their casing: `### Understanding the POST request`, `## SWML basics`, `## **Parameters**`.
- The fixed reference section names (`## **Parameters**`, `## **Properties**`, `## **Returns**`,
  `## **Examples**` / `## **Example**`) are bolded and used verbatim вҖ” these are the house names.

### 5.1 Reference page

For a single symbol: an application, API/CLI command, function, module, or channel variable. Keep it terse and lookup-oriented (~50вҖ“80 lines).

**Structure (in order, dropping any section the source doesn't support):**
1. One- or two-sentence description (no heading вҖ” it follows the front matter directly).
2. Optional callout (`<Note>` / `<Warning>`) for a critical prerequisite or gotcha.
3. The interface-shape section вҖ” use the heading that fits: `## **Parameters**` (function/command arguments), `## **Properties**` (data object / struct), or `## Attributes` (XML element). One `<ParamField>` per element; wrap nested fields in `<Indent>`. See [Describing an interface shape](#describing-an-interface-shape).
4. `## **Returns**` вҖ” only if the symbol returns something meaningful.
5. `## **Examples**` (or `## **Example**` for one) вҖ” runnable code in `<CodeBlocks>` or a single fenced block.

**Skeleton:**

````mdx
---
title: "originate"
slug: /reference/api-commands/originate
description: Originate a new call from FreeSWITCH to a destination and bridge it to a dialplan target.
max-toc-depth: 3
---

Originate a new call from FreeSWITCH to a destination and bridge it to a dialplan
extension, application, or another endpoint.

<Note>
`originate` is a foreground command on the API/CLI; the channel executes synchronously
unless you prefix the dial string with `{origination_caller_id_number=...}` options.
</Note>

## **Parameters**

<ParamField path="dial_string" type="string" required={true} toc={true}>
  The endpoint to call, e.g. `sofia/gateway/my_gateway/15551234567`.
</ParamField>

<ParamField path="application" type="string" required={true} toc={true}>
  The dialplan application (or extension) to execute once the call is answered.
</ParamField>

<ParamField path="options" type="object" toc={true}>
  Channel variables passed inline in `{ }` before the dial string.
</ParamField>

<Indent>

<ParamField path="options.origination_caller_id_number" type="string" toc={true}>
  Caller ID number presented to the destination.
</ParamField>

<ParamField path="options.call_timeout" type="integer" default="60" toc={true}>
  Seconds to wait for the call to be answered before giving up.
</ParamField>

</Indent>

## **Returns**

`+OK <uuid>` on success, where `<uuid>` is the new channel's unique ID.
`-ERR <reason>` on failure.

## **Example**

```bash
originate {origination_caller_id_number=15550001111}sofia/gateway/my_gateway/15551234567 &playback(/tmp/welcome.wav)
```
````

> **Multi-format / multi-language examples** go in `<CodeBlocks>` вҖ” see [В§6](#codeblocks--codeblock) and the worked example in [В§11](#11-worked-examples).

#### Describing an interface shape

Every reference page documents an **interface** вҖ” the shape a user fills in to invoke the thing.
The surface syntax differs (a JSON/YAML data object, a function signature in Python / TypeScript /
Go / Ruby / C++, an XML element's attributes, a callback), but **the documentation pattern is
always the same:**

> Describe the shape with `<ParamField>` вҖ” **one per element, in source order, showing the
> hierarchy** вҖ” then follow with a **practical code example in that interface's native syntax**.

What changes per interface is only the section heading, what a field maps to, and the `path` form:

| Interface | A `<ParamField>` describesвҖҰ | Heading | `path` form |
|-----------|------------------------------|---------|-------------|
| Data object вҖ” JSON/YAML (e.g. SWML) | a key in the object | `## **Properties**` | dot-notation (`answer.max_duration`) |
| Function / method вҖ” Python, TS, Go, Ruby, C++ | an argument | `## **Parameters**` | bare arg name (`timeout`) |
| XML element вҖ” dialplan / config | an element attribute | `## Attributes` | bare attribute name (`data`) |
| Callback / event handler | a callback argument or payload field | `## **Parameters**` | arg name; dot-notation for payload fields |

Shared rules for all of them:
- **One `<ParamField>` per element**, in the order the source defines them.
- **Show hierarchy** with dot-notation `path` and wrap nested fields in `<Indent>` ([В§6](#indent)).
- **Write `type` in the interface's own terms** вҖ” `integer` / `string[]` for SWML, `Optional[List[str]]`
  for Python, `string | undefined` for TypeScript, `*int` for Go, etc. Keep names in source casing
  (`destination_number`, `answerOnBridge`).
- **End with a practical example** in native syntax вҖ” `<CodeBlocks>` when the same call has multiple
  forms (YAML + JSON, or several languages), a single fenced block otherwise.
- For a field with a fixed value set, list the values in its description, or вҖ” if long вҖ” in an
  anchored sub-section (`#### Values for \`record\`  [#record-values]`) with a table.

**The pattern across a few interfaces:**

*Data object вҖ” SWML (JSON/YAML), nested keys via dot-notation + `<Indent>`:*
```mdx
## **Properties**

<ParamField path="answer" type="object" toc={true}>
  An object that accepts the following properties.
</ParamField>

<Indent>

<ParamField path="answer.max_duration" type="integer" default="14400" toc={true}>
  Maximum duration in seconds for the call.
</ParamField>

</Indent>
```

*Function / method вҖ” type written in the language's own terms (here Python; TS/Go/Ruby/C++ differ only in `type`):*
```mdx
## **Parameters**

<ParamField path="hint" type="str" required={true} toc={true}>
  A word or phrase to boost recognition accuracy.
</ParamField>
```

*XML element вҖ” one field per attribute, bare attribute name as `path`:*
```mdx
## Attributes

<ParamField path="application" type="string" required={true} toc={true}>
  The dialplan application to execute, e.g. `playback`, `bridge`, `set`.
</ParamField>
```

*Callback / event handler вҖ” describe the handler's arguments; document payload fields inline (or via dot-notation when strongly typed):*
```mdx
## **Parameters**

<ParamField path="handler" type="Callable[[str, dict], None]" required={true} toc={true}>
  Callback invoked on each event, with signature `(event_type, data)`.

  - `event_type` вҖ” the event label string
  - `data` вҖ” full payload including `call_id` and event-specific fields
</ParamField>
```

In every case, a `## **Examples**` section in the interface's native syntax follows.

### 5.2 Section overview page (`overview.mdx` or `index.mdx`)

A section's landing page. Orients the reader and links to the section's pages. The **filename
follows [Section landing pages](#section-landing-pages)**: `overview.mdx` for a top-level section,
`index.mdx` for a nested topic subsection. The page content is the same either way.

**Structure:**
1. Intro paragraph(s) вҖ” what this area is and when you'd use it.
2. Optional core-concept `##` sections.
3. `<CardGroup>` / `<Cards>` linking the child pages.

**Skeleton** (saved as `overview.mdx` here, since "API Commands" is a top-level section; `sidebar-title: Overview` + `position: 0` make it the section's first child):

```mdx
---
title: "API Commands"
sidebar-title: Overview
subtitle: Control FreeSWITCH at runtime from the CLI, event socket, or HTTP.
slug: /reference/api-commands
description: Reference for FreeSWITCH API commands available via fs_cli, ESL, and mod_xml_rpc.
max-toc-depth: 3
position: 0
---

API commands let you query and control a running FreeSWITCH instance вҖ” originating
calls, inspecting channels, and reloading configuration вҖ” without restarting.

## Common commands

<CardGroup cols={3}>
  <Card title="originate" href="/reference/api-commands/originate">
    Originate a new outbound call.
  </Card>
  <Card title="uuid_bridge" href="/reference/api-commands/uuid-bridge">
    Bridge two existing channels by UUID.
  </Card>
  <Card title="reloadxml" href="/reference/api-commands/reloadxml">
    Reload the XML configuration.
  </Card>
</CardGroup>
```

### 5.3 Guide / tutorial page

Task-oriented and narrative. Walks the reader through accomplishing something.

**Structure:**
1. Intro paragraph вҖ” what the reader will accomplish.
2. Optional `<Info>` / `<Tip>` orientation callout.
3. Body: prose + `<CodeBlocks>`; use `<Steps>` for ordered procedures.
4. `## Next steps` вҖ” a bullet list of links (or `<Cards>`).

**Skeleton:**

```mdx
---
title: Route inbound calls with the XML dialplan
subtitle: Match inbound calls and send them to applications using the XML dialplan.
slug: /guides/inbound-routing
description: Learn how to route inbound calls to dialplan applications using FreeSWITCH XML dialplan conditions.
max-toc-depth: 3
---

This guide shows how to match inbound calls and route them to dialplan applications
using FreeSWITCH XML dialplan conditions.

<Info>
For the full set of dialplan applications you can call from a condition, see the
[Applications reference](/reference/applications).
</Info>

## Add a dialplan extension

<Steps>

### Create the extension

Add an `<extension>` block to `conf/dialplan/default.xml`.

### Match the destination number

Use a `<condition>` on `destination_number` with a regular expression.

### Execute applications

Add `<action>` elements for each application to run on a match.

</Steps>

```xml
<extension name="inbound_main">
  <condition field="destination_number" expression="^(15551234567)$">
    <action application="answer"/>
    <action application="playback" data="/tmp/welcome.wav"/>
  </condition>
</extension>
```

## Next steps

- **[Applications reference](/reference/applications)**: Every application you can call from a dialplan action.
- **[Channel variables](/reference/channel-variables)**: Variables you can read and set during routing.
```

## 6. Component catalog

These are the components available to you. Use **only** these. At a glance:

| Component | Use it when you need toвҖҰ |
|-----------|--------------------------|
| `ParamField` | Document a single parameter, property, or field. |
| `Indent` | Nest an object's sub-fields under their parent param. |
| `CodeBlocks` / `CodeBlock` | Show the same example in multiple formats/languages as tabs. |
| `Tabs` / `Tab` | Offer switchable non-code variants (e.g. two parameter shapes). |
| `Note` / `Tip` / `Warning` / `Info` | Call out a prerequisite, suggestion, hazard, or pointer. |
| `Accordion` / `AccordionGroup` | Hide a long example or optional aside until expanded. |
| `Card` / `CardGroup` / `Cards` | Link related pages in a grid (overviews, "Next steps"). |
| `Steps` / `Step` | Lay out an ordered procedure in a guide. |
| `Frame` | Wrap an image or diagram with a caption. |
| `Markdown` | Include a shared, reusable snippet fragment. |
| `Button` | Add a call-to-action link on a landing/getting-started page. |
| `Tooltip` | Define a jargon term inline on hover. |
| `Icon` | Place an inline icon. |

Each entry below gives when to use it, a correct example, and what not to do. All examples obey [В§7](#7-strict-mdx-v3-rules).

### ParamField

Describe one element of an interface вҖ” a function argument, a data-object key, an XML attribute,
or a callback field. One per element. This is the workhorse component for every reference page; it
describes the **shape** of whatever interface the page documents, regardless of language or format.
See [Describing an interface shape](#describing-an-interface-shape) for the per-interface pattern.

```mdx
<ParamField path="call_timeout" type="integer" required={true} default="60" toc={true}>
  Seconds to wait for the call to be answered before giving up.
</ParamField>
```

- `path` вҖ” the element's name. Use the **bare name** for a top-level argument or an XML/data
  attribute (`path="data"`, `path="timeout"`); use **dot-notation** for a nested field of an object
  (`path="originate.options.call_timeout"`). Keep the native casing exactly as the source uses it
  (e.g. `destination_number`, `answerOnBridge`).
- `type` вҖ” the source type: `string`, `integer`, `boolean`, `object`, `array`, `float`, or a language-accurate type.
- `required={true}` вҖ” include **only** for required params; omit entirely for optional.
- `default="..."` вҖ” include **only** when the source defines a default.
- `toc={true}` вҖ” add when the param should get a table-of-contents anchor (use on reference params).
- **DON'T** write `required={false}` вҖ” just omit the attribute. **DON'T** invent a `default`.

### Indent

Wrap the sub-fields of an object parameter to nest them visually under their parent.

```mdx
<ParamField path="options" type="object" toc={true}>
  Inline channel variables.
</ParamField>

<Indent>

<ParamField path="options.call_timeout" type="integer" default="60" toc={true}>
  Seconds to wait before giving up.
</ParamField>

</Indent>
```

- **DON'T** forget the blank lines inside `<Indent>` вҖ” MDX needs them to parse the nested components.

### CodeBlocks / CodeBlock

A set of related code examples shown as switchable tabs вҖ” use for **the same example in multiple formats/languages** (YAML + JSON, Python + TypeScript, etc.).

````mdx
<CodeBlocks>
<CodeBlock title="YAML">
```yaml
version: 1.0.0
sections:
  main:
    - answer: {}
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": { "main": [ { "answer": {} } ] }
}
```
</CodeBlock>
</CodeBlocks>
````

- For a **single** example, just use a plain fenced code block with a language tag вҖ” no `<CodeBlocks>` wrapper needed.
- **DON'T** omit the `title` on a `<CodeBlock>`. **DON'T** omit the language tag on the fence.

### Tabs / Tab

Switchable content blocks for non-code alternatives вҖ” e.g. two parameter shapes, or platform variants.

```mdx
<Tabs>
<Tab title="Single URL">
  Content for the first variant.
</Tab>
<Tab title="Multiple URLs">
  Content for the second variant.
</Tab>
</Tabs>
```

### Note / Tip / Warning / Info

Callouts. Pick by intent:

- `<Note>` вҖ” important supporting information or a prerequisite.
- `<Tip>` вҖ” a helpful suggestion or best practice.
- `<Warning>` вҖ” something that can break or cause data loss / billing.
- `<Info>` вҖ” neutral pointer to related reference material.

```mdx
<Warning>
Reloading the XML config drops in-progress originations that depend on changed gateways.
</Warning>
```

- An optional title is supported: `<Warning title="Calling only">...`.
- **DON'T** stack three callouts in a row вҖ” fold them into one or move detail into prose.

### Accordion / AccordionGroup

Collapsible content. Use to hide a long example or an optional aside until the reader expands it.

```mdx
<AccordionGroup>
<Accordion title="See the full configuration">
  ```xml
  <configuration name="sofia.conf">...</configuration>
  ```
</Accordion>
</AccordionGroup>
```

### Card / CardGroup / Cards

Navigation grids. Use on overview/landing pages and "Next steps" sections to link related pages.

```mdx
<CardGroup cols={3}>
  <Card title="originate" href="/reference/api-commands/originate">
    Originate a new outbound call.
  </Card>
  <Card title="uuid_bridge" href="/reference/api-commands/uuid-bridge">
    Bridge two channels by UUID.
  </Card>
</CardGroup>
```

- `cols={2}` or `cols={3}` (note the braces вҖ” it's a number).
- `<Cards>` is the same idea without an explicit column count.
- Every `href` must resolve to a real page ([В§8](#8-linking-code-and-snippets)).

### Steps

An ordered procedure in a guide. Each `###` heading inside becomes a numbered step.

```mdx
<Steps>

### Create the extension

Add an `<extension>` block.

### Match the number

Add a `<condition>`.

</Steps>
```

- **DON'T** omit the blank lines around the `###` headings вҖ” they're parsed as Markdown.

An explicit `<Step title="...">` form is also supported when you want a step to contain
mixed components rather than a single heading + prose:

```mdx
<Steps>
<Step title="Install the package">
  ```bash
  apt-get install freeswitch
  ```
</Step>
<Step title="Start the service">
  Run `systemctl start freeswitch` and confirm it's listening.
</Step>
</Steps>
```

Use one form or the other consistently within a page вҖ” don't mix `###` steps and `<Step>` steps.

### Button

A call-to-action link, used on landing / getting-started pages. Not for reference pages.

```mdx
<Button intent="primary" href="/guides/get-started" rightIcon="arrow-right">Get started</Button>
```

- `intent` (`primary`, etc.), `href`, and optional `rightIcon`/`leftIcon`.

### Tooltip

Inline hover definition for a piece of jargon. Use sparingly, in prose.

```mdx
<Tooltip tip="Automatic Speech Recognition, also known as Speech-to-Text.">ASR</Tooltip>
```

### Frame

Wrap an image or diagram, with an optional caption.

```mdx
<Frame caption="Call routing overview">
  <img src="/assets/images/freeswitch/routing-diagram.webp" alt="Inbound call routing flow" />
</Frame>
```

- `<img>` must be **self-closed** (`/>`) and have meaningful `alt` text.

### Markdown (snippet include)

Pull in a shared, reusable fragment. Use only if you actually have a shared snippet to include.

```mdx
<Markdown src="/snippets/freeswitch/_channel-variable-table.mdx" />
```

- Self-closing. **DON'T** invent snippet paths вҖ” only reference a partial you've actually created.

### Icon

An inline icon, by Font Awesome slug.

```mdx
<Icon icon="circle-info" size="4" />
```

## 7. Strict MDX v3 rules

The site compiles MDX 3 with Acorn JSX parsing. The following are **build-breaking errors**, not style preferences. Self-check every one before emitting.

| Rule | вң… Do | вқҢ Don't |
|------|-------|---------|
| **Close every tag** | `<br />`, `<hr />`, `<img src="x.webp" alt="y" />` | `<br>`, `<img src="x.webp">` |
| **Self-close void/standalone components** | `<Markdown src="..." />` | `<Markdown src="...">` |
| **Comments are JSX expressions** | `{/* a comment */}` | `<!-- a comment -->` |
| **String attrs in quotes** | `title="YAML"` | `title=YAML` |
| **Boolean / number / expression attrs in braces** | `required={true}`, `cols={3}`, `toc={true}` | `required="true"`, `cols=3` |
| **Escape bare `<` and `{` in prose** | `` use `<` for less-than ``, `` the `{ }` block `` | `use < for less-than`, `the { } block` (Acorn tries to parse these as JSX/expressions) |
| **Blank line between a block component and adjacent Markdown** | `</Note>`вҸҺвҸҺ`## Next` | `</Note>`вҸҺ`## Next` |
| **Blank lines inside `<Indent>`, `<Steps>`, `<Accordion>`** so inner Markdown/components parse | (see В§6 examples) | packing children with no surrounding blank lines |
| **Fence every code block with a language** | ` ```bash `, ` ```xml `, ` ```python ` | ` ``` ` with no language |
| **No stray 4-space indentation** in prose/JSX (it becomes an indented code block) | left-align body text | indenting paragraphs under a component |
| **Headings: use `##`+, never a body `# H1`** | `## **Parameters**` | `# originate` in the body |
| **Internal links must resolve** | `/reference/api-commands/originate` | a path with no page behind it |

**The single most common failure:** a literal `<` or `{` sitting in prose (e.g. describing C syntax, a template, or a comparison). If you must show it inline, wrap it in backticks; if it's a block, put it in a fenced code block.

## 8. Linking, code, and snippets

**Internal links.** Use site-absolute paths that match a page's `slug`. Two equivalent styles are in use:

- Inline: `[originate](/reference/api-commands/originate)`
- Reference-style (define near the top of the file, reference by label) вҖ” preferred when the same target is linked several times:

```mdx
[originate]: /reference/api-commands/originate

See [`originate`][originate] for the full parameter list.
```

Every link must resolve to a real `slug`, and any `#anchor` must match a heading that has a TOC anchor (`toc={true}` params or `##`/`###` headings). Broken links fail the build.

**Code examples.** Fence rules live in [В§7](#7-strict-mdx-v3-rules) and `<CodeBlocks>` in [В§6](#codeblocks--codeblock). Beyond those: highlight key lines with `{n}` after the language (` ```python {7} `), and keep examples realistic and runnable вҖ” derive them from the source's actual usage.

**Snippets.** Factor content into a shared `<Markdown src="..." />` partial ([В§6](#markdown-snippet-include)) only when it's genuinely reused across pages (e.g. a channel-variable legend); otherwise inline it.

## 9. Pre-output self-check (definition of done)

Before you emit any page, verify **every** item. Fix misses; do not emit a page with a known miss.

**Front matter**
- [ ] `title`, `slug`, `description`, `max-toc-depth: 3` all present.
- [ ] `slug` is present, lowercase-kebab, short/descriptive, and **not** a copy of the folder path (drops filing folders); no product prefix, no extension.
- [ ] No fabricated `id`/UUID. No body `# H1`.
- [ ] `overview.mdx` pages set `position: 0`; other pages leave `position` off (alphabetical) unless a deliberate order is required.

**Structure**
- [ ] Page type identified; sections match its skeleton order ([В§5](#5-page-types-and-skeletons)).
- [ ] If this is a section landing page, the filename is correct: `overview.mdx` (top-level) vs `index.mdx` (nested topic) vs none for pure organization ([В§3](#section-landing-pages)).
- [ ] Body opens with the description paragraph, then `##` sections.
- [ ] Headings are **sentence case** (not title case); only first word + proper nouns/acronyms/identifiers capitalized.
- [ ] No empty/placeholder sections (dropped any with no source basis).

**Source fidelity**
- [ ] Every parameter, type, default, and return value is traceable to the source. Nothing invented.
- [ ] `required={true}` only on truly-required params; `default` only where the source defines one.

**MDX v3 ([В§7](#7-strict-mdx-v3-rules))**
- [ ] All tags closed; standalone components self-closed.
- [ ] No `<!-- -->` comments.
- [ ] String attrs quoted; boolean/number/expression attrs braced.
- [ ] No bare `<` or `{` in prose.
- [ ] Blank lines around block components and inside `<Indent>`/`<Steps>`/`<Accordion>`.
- [ ] Every code fence has a language tag.

**Components & links**
- [ ] Only catalog components used ([В§6](#6-component-catalog)).
- [ ] Every `href`/link resolves to a real slug; anchors match real headings.

## 10. Anti-patterns (common failures to avoid)

- **Inventing data** вҖ” adding a `default`, type, or parameter the source doesn't define. The fastest way to ship a wrong doc.
- **A body `# H1`** вҖ” duplicates the title the front matter already renders.
- **Title-case headings** ("Next Steps", "Play A Local File") вҖ” use sentence case ("Next steps", "Play a local file").
- **`required="false"`** вҖ” never; omit the attribute for optional params.
- **HTML comments / unclosed `<img>`/`<br>`** вҖ” instant build failure under MDX 3.
- **Bare `<` or `{` in prose** вҖ” wrap in backticks or a code fence.
- **Wrapping a single example in `<CodeBlocks>`** вҖ” only multi-format/multi-language examples need it.
- **Placeholder sections** ("Parameters: None") вҖ” drop the section instead.
- **New section names** ("Arguments", "Output") вҖ” use the house names: **Parameters** / **Properties** / **Attributes** (XML elements) / **Returns** / **Examples**.
- **Wrong section landing file** вҖ” using `index.mdx` for a top-level section (makes the header a link when it shouldn't be) or `overview.mdx` for a nested topic subsection. Top-level вҶ’ `overview.mdx`; nested topic вҶ’ `index.mdx`; pure organization вҶ’ no landing file ([В§3](#section-landing-pages)).
- **Guessing links** вҖ” if you reference a page, make sure it (will) exist at that slug.

### 11.1 Reference page вҖ” a dialplan application

````mdx
---
title: "playback"
slug: /reference/applications/playback
description: Play an audio file or TTS prompt to the calling channel.
max-toc-depth: 3
---

Play an audio file (or a TTS prompt) to the calling channel. The channel must be
answered or in early media before playback begins.

<Note>
Playback is interruptible by DTMF only when the `playback_terminators` channel
variable is set. Otherwise the file plays to completion.
</Note>

## **Parameters**

<ParamField path="file" type="string" required={true} toc={true}>
  Path or URI of the sound file to play. Supports local paths, `http://` URLs,
  and TTS via the `say:` and `tts://` prefixes.
</ParamField>

<ParamField path="options" type="object" toc={true}>
  Channel variables that modify playback, set before the application runs.
</ParamField>

<Indent>

<ParamField path="options.playback_terminators" type="string" default="none" toc={true}>
  DTMF digits that stop playback, e.g. `#` or `*0`. Set to `none` to disable.
</ParamField>

<ParamField path="options.playback_delimiter" type="string" toc={true}>
  Delimiter used when playing a list of files in a single `file` argument.
</ParamField>

</Indent>

## **Returns**

Sets the `playback_last_offset_pos` channel variable to the byte offset at which
playback ended, and `playback_terminator_used` to the DTMF digit that stopped it
(if any).

## **Examples**

### Play a local file

```xml
<action application="playback" data="/usr/share/freeswitch/sounds/welcome.wav"/>
```

### Play with an interrupt key

```xml
<action application="set" data="playback_terminators=#"/>
<action application="playback" data="ivr/ivr-welcome_to_freeswitch.wav"/>
```
````

### 11.2 Guide page вҖ” a short tutorial

````mdx
---
title: Originate an outbound call from the CLI
subtitle: Place a test call from fs_cli and bridge it to an application.
slug: /guides/originate-from-cli
description: Learn how to place an outbound call from the FreeSWITCH CLI and bridge it to a dialplan application.
max-toc-depth: 3
---

This guide walks through placing an outbound call from `fs_cli` and bridging the
answered call to a dialplan application вҖ” useful for quickly testing a gateway.

<Tip>
Run `sofia status gateway my_gateway` first to confirm the gateway is registered
before you try to originate through it.
</Tip>

## Place the call

<Steps>

### Open the CLI

Connect to the running FreeSWITCH instance:

```bash
fs_cli
```

### Originate and bridge

Call the destination through your gateway and play a file once answered:

```bash
originate sofia/gateway/my_gateway/15551234567 &playback(/tmp/welcome.wav)
```

### Confirm the result

A successful origination returns `+OK <uuid>`. A failure returns `-ERR` with a
hangup cause such as `NORMAL_TEMPORARY_FAILURE`.

</Steps>

## Set caller ID

Prefix the dial string with inline channel variables to control caller ID:

```bash
originate {origination_caller_id_number=15550001111}sofia/gateway/my_gateway/15551234567 &echo
```

## Next steps

- **[`originate` reference](/reference/api-commands/originate)**: Every option and return value.
- **[Channel variables](/reference/channel-variables)**: Variables you can set inline on the dial string.
````
