Building an agent that holds up in production takes more than a good prompt.
This guide collects the practices that matter across the whole design: how work divides between
the prompt and your code, how to write for a real-time voice medium, how to help speech recognition,
and how to test, observe, and stay compliant once callers arrive.

## Split the work between the prompt and your code

The single highest-leverage design decision is what you *don't* put in the prompt.

Language models are excellent at conversation: tone, comprehension, extracting details from natural speech.
They are unreliable at arithmetic, current facts, and rule-following under pressure.
A prompt is a suggestion; code is a constraint.
Give each side the work it's good at.
The prompt covers who the agent is, how it speaks, what it's there to accomplish,
and when to reach for its functions.
Your code, behind [SWAIG functions][tool-calling] (SignalWire's tool calling), covers prices,
inventory, calculations, policy, and anything else with a right answer.

<Tip>
A useful rule of thumb: if the answer must be exact tomorrow, it can't live in the prompt.
</Tip>

### Keep business data out of the prompt

It's tempting to paste the catalog, the price list, or the policy manual into the prompt.
Resist it.
The prompt is fixed when the call starts, but your data keeps changing;
a function call reads the current value at the moment the caller asks.
Data in the prompt is *recalled* by the model, approximately,
while data from a function call is *returned* by your code, exactly.
And the model has no way to tell your pasted policy apart from anything else it has read.
Your backend is the system of record.

Move that data behind SWAIG functions: a lookup backed by your webhook or the Server SDK,
or a serverless [DataMap][swml-datamap] for straightforward API calls and pattern-matched responses.

## Write the prompt for conversation, not logic

With business logic out of the way, the prompt's job is focused: define the agent's identity,
its conversational duties, and its boundaries.
Outline those necessities clearly, then stop.
Overloading the AI with instructions muddles its behavior rather than tightening it,
and since the prompt is part of every conversational turn, a brief prompt is also cheaper to run.
See [avoiding over-prompting][prompt-eng-best-practices] for techniques.

Structure the prompt with [Markdown](https://www.markdownguide.org/basic-syntax/):
headings and lists keep it organized and legible,
narrowing how the AI interprets each section.

<CodeBlocks>
<CodeBlock title="SWML">
```yaml
version: 1.0.0
sections:
  main:
    - ai:
        prompt:
          text: |
            Your name is Sam. You are the phone assistant for Hartwell Hardware.

            ## Personality and duties
            You are friendly and practical. Help callers check whether an item
            is in stock and what it costs, using the functions available to you.

            ## Greeting rules
            Greet the caller, introduce yourself as Sam, and ask what item
            they're looking for. Prefix the greeting with 'good morning',
            'good afternoon', or 'good evening' depending on the time of day.
          temperature: 0.89
          top_p: 0.64
```
</CodeBlock>
<CodeBlock title="Server SDK (Python)">
```python
from signalwire import AgentBase

agent = AgentBase(name="hartwell-hardware")

agent.prompt_add_section(
    "Role",
    "Your name is Sam. You are the phone assistant for Hartwell Hardware."
)
agent.prompt_add_section(
    "Personality and duties",
    "You are friendly and practical. Help callers check whether an item "
    "is in stock and what it costs, using the functions available to you."
)
agent.prompt_add_section(
    "Greeting rules",
    "Greet the caller, introduce yourself as Sam, and ask what item "
    "they're looking for. Prefix the greeting with 'good morning', "
    "'good afternoon', or 'good evening' depending on the time of day."
)
agent.set_params({"temperature": 0.89, "top_p": 0.64})

if __name__ == "__main__":
    agent.run()
```
</CodeBlock>
</CodeBlocks>

Notice what this prompt *doesn't* contain: no product catalog, no prices, no stock counts.
That information lives in the store's inventory system behind the agent's functions,
so the prompt stays short and the answers stay accurate.

Some context does belong in the prompt: stable facts like business hours, or the agent's scope.
Place dynamic or per-call values (data interpolated into the prompt at request time)
at the **bottom** of the prompt, so the stable part stays identical from turn to turn and from call to call.

For the full craft of prompt writing, including structure, examples, and iterative refinement,
see the [prompt engineering guide][prompt-engineering].

### Tune the model parameters

The [`prompt` object][ai-prompt-reference] accepts parameters that shape the AI's responses:

- **Temperature (`temperature`)**: influences the randomness of the AI's output.
  Values closer to 0 make responses more deterministic and focused; higher values introduce variety.
- **Top P (`top_p`)**: also controls diversity. A lower value narrows the range of potential responses.
- **Confidence (`confidence`)**: the threshold for speech-detection events.
  A lower value shortens the pause after the caller speaks, but may cause false positives.
- **Presence penalty (`presence_penalty`)**: positive values make the AI more likely to introduce new topics.
- **Frequency penalty (`frequency_penalty`)**: positive values discourage repetition.

## Design for real-time voice

Voice AI adds a dimension that text chat doesn't have: time.
Two concepts matter when tuning how your agent converses.
**End-pointing** (end-of-utterance detection) is the AI's ability to detect when the caller has finished speaking,
and **turnaround** (turn-taking) is the time it takes the AI to process input and begin its response.

Neither can be instant.
Natural language processing takes time: the AI analyzes syntax, semantics, and context,
even if only for a fraction of a second.
Real conversation is full of pauses, overlaps, and changes of direction that make "finished speaking" ambiguous.
And generating a coherent, contextually appropriate response is computational work.

<Note>
No system delivers truly instantaneous end-pointing and turnaround.
Judge latency on live calls with your own configuration rather than on recorded demos,
and measure the interval the caller experiences: from the moment they stop talking
to the moment they hear the reply begin.
Consistency matters as much as the average; a steady conversation feels better
than a fast one with occasional stalls.
</Note>

Rather than chasing zero, make the wait feel natural.
Filler phrases give the AI something to say while a function call is running;
configure them [per-function](/docs/swml/reference/calling/ai/swaig/functions) and
[agent-wide](/docs/swml/reference/calling/ai/languages).
Background audio, such as typing or office ambience, provides immediate and familiar feedback
while processing happens; set it with [`params.background_file`](/docs/swml/reference/calling/ai/params).
And keep your function handlers fast: your webhook's latency is part of the conversation,
so move slow work out of band and respond promptly.

## Help speech recognition with hints

Hints boost recognition accuracy for the words that matter in your domain,
so "caulk" doesn't arrive as "chalk."

Introduce the vocabulary your agent should expect, such as product names, jargon, and place names:

<CodeBlocks>
<CodeBlock title="SWML">
```yaml
version: 1.0.0
sections:
  main:
    - ai:
        prompt:
          text: |
            Your name is Sam. You are the phone assistant for Hartwell Hardware.
            Help callers check whether items are in stock.
        hints:
          - caulk
          - torx
          - galvanized
          - PVC
          - primer
          - spackle
```
</CodeBlock>
<CodeBlock title="Server SDK (Python)">
```python
from signalwire import AgentBase

agent = AgentBase(name="hartwell-hardware")

agent.prompt_add_section(
    "Role",
    "Your name is Sam. You are the phone assistant for Hartwell Hardware. "
    "Help callers check whether items are in stock."
)
agent.add_hints(["caulk", "torx", "galvanized", "PVC", "primer", "spackle"])

if __name__ == "__main__":
    agent.run()
```
</CodeBlock>
</CodeBlocks>

See [`hints`][ai-reference] in the SWML reference, or [speech recognition hints][sdk-hints] in the Server SDK.

## Test, monitor, and iterate

Test with real users before deploying.
Scripted test calls follow the script; customers won't, and beta callers surface the gaps.

Once callers arrive, review real calls.
Define a `post_prompt` and set a [`post_prompt_url`][ai-reference] to receive a report after
each call (summary, conversation log, and every function call), and read the transcripts
where the agent hesitated or chose the wrong function.
That's where the next fix comes from.
Usage metrics fill in the aggregate picture:
token consumption, interaction times, and call outcomes reveal both cost and quality trends.

An agent is never finished.
Your business changes, so revisit the prompt and functions periodically,
and check the documentation and release notes for new parameters, voices,
and features your agent can use as SignalWire's AI capabilities evolve.

## Stay compliant with regulations

The FCC has ruled that calls using AI-generated voices fall under the
[Telephone Consumer Protection Act (TCPA)](https://www.fcc.gov/sites/default/files/tcpa-rules.pdf):
**unsolicited** robocalls made with AI voices are illegal without the receiver's prior consent.

This does **not** make AI agents illegal. It means they must be used responsibly and transparently:

- Clearly state that the caller is interacting with an AI agent.
- Provide a simple, immediate way for the caller to leave the AI interaction.
- For outbound use especially, ensure recipients have consented to interact with an AI.

## Putting it together

Here is Sam as a complete script, with each practice in place:
a lean prompt, stock and prices served by the store's inventory system instead of pasted text,
a filler phrase to cover the lookup, hints for domain vocabulary, and a post-call
report for review.

<CodeBlocks>
<CodeBlock title="SWML">
```yaml
version: 1.0.0
sections:
  main:
    - ai:
        prompt:
          text: |
            Your name is Sam. You are the phone assistant for Hartwell Hardware.

            ## Personality and duties
            You are friendly and practical. Help callers check whether an item
            is in stock and what it costs. Look items up with check_stock, and
            only quote prices and quantities that check_stock returned.

            ## Greeting rules
            Greet the caller, introduce yourself as Sam, and ask what item
            they're looking for.
          temperature: 0.89
          top_p: 0.64
        SWAIG:
          functions:
            - function: check_stock
              description: Look up an item in the store inventory system for current stock and price
              parameters:
                type: object
                properties:
                  item:
                    type: string
                    description: The item the caller asked about, in their own words
                required:
                  - item
              fillers:
                en-US:
                  - Let me check the shelf for you...
              web_hook_url: https://example.com/inventory
        hints:
          - caulk
          - torx
          - galvanized
          - PVC
          - primer
          - spackle
        post_prompt:
          text: Summarize the call and provide the summary in JSON format.
        post_prompt_url: https://example.com/post-prompt
```
</CodeBlock>
<CodeBlock title="Server SDK (Python)">
```python
from signalwire import AgentBase, FunctionResult

# Stand-in for your inventory system: a database query, an internal API call
INVENTORY = {
    "galvanized wood screws": {"sku": 4471, "stock": 14, "price": 8.49},
    "pvc primer": {"sku": 2205, "stock": 9, "price": 6.99},
    "caulk gun": {"sku": 3118, "stock": 5, "price": 12.99},
}

class HardwareAgent(AgentBase):
    def __init__(self):
        super().__init__(name="hartwell-hardware")
        self.add_language("English", "en-US", "rime.spore")

        self.prompt_add_section(
            "Role",
            "Your name is Sam. You are the phone assistant for Hartwell Hardware."
        )
        self.prompt_add_section(
            "Personality and duties",
            "You are friendly and practical. Help callers check whether an item "
            "is in stock and what it costs. Look items up with check_stock, and "
            "only quote prices and quantities that check_stock returned."
        )
        self.prompt_add_section(
            "Greeting rules",
            "Greet the caller, introduce yourself as Sam, and ask what item "
            "they're looking for."
        )
        self.set_params({"temperature": 0.89, "top_p": 0.64})

        self.add_hints(["caulk", "torx", "galvanized", "PVC", "primer", "spackle"])
        self.set_post_prompt("Summarize the call and provide the summary in JSON format.")

        self.define_tool(
            name="check_stock",
            description="Look up an item in the store inventory system for current stock and price",
            parameters={
                "type": "object",
                "properties": {
                    "item": {
                        "type": "string",
                        "description": "The item the caller asked about, in their own words"
                    }
                },
                "required": ["item"]
            },
            handler=self.check_stock,
            fillers={"en-US": ["Let me check the shelf for you..."]}
        )

    def check_stock(self, args, raw_data):
        item = args.get("item", "").lower().strip()
        match = INVENTORY.get(item)
        if not match:
            return FunctionResult(
                f"No item matching '{item}'. Ask the caller to describe it differently."
            )
        return FunctionResult(
            f"Item {match['sku']}, {item}: {match['stock']} in stock "
            f"at ${match['price']:.2f}. Offer to set one aside at the front desk."
        )

if __name__ == "__main__":
    agent = HardwareAgent()
    agent.run()
```
</CodeBlock>
</CodeBlocks>

When a caller asks about wood screws, the platform sends an HTTP POST to your function's endpoint
(the `web_hook_url` in SWML, or the endpoint the Server SDK serves for you):

```json
{
  "function": "check_stock",
  "argument": {
    "parsed": [{ "item": "galvanized wood screws" }]
  }
}
```

Your server queries the inventory database (the same one the registers use) and replies:

```json
{
  "response": "Item 4471, galvanized wood screws: 14 in stock at $8.49. Offer to set one aside at the front desk."
}
```

Sam relays the answer in its own voice and offers to hold a box.
Every number in that sentence came out of the inventory system during the call.
When the price changes tomorrow, the agent is already right,
and the post-call report shows you every lookup it made.
