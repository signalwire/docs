# `main`

Example: step\_function\_inheritance\_demo

This example exists to teach one specific gotcha: the per-step functions whitelist INHERITS from the previous step when omitted.

Why this matters ---------------- A common mistake when building multi-step agents is to assume each step starts with a fresh tool set. It does not. The runtime only resets the active set when a step explicitly declares its `functions` field. If you forget SetFunctions() on a later step, the previous step's tools quietly remain available.

This file shows four step-shaped patterns side by side:

1. step\_lookup — explicitly whitelists `lookup_account`
2. step\_inherit — has NO SetFunctions() call. Inherits step\_lookup's whitelist, so `lookup_account` is still callable here. This is rarely what you want.
3. step\_explicit — explicitly whitelists `process_payment`. The previously inherited `lookup_account` is now disabled, and only `process_payment` is active.
4. step\_disabled — explicitly disables ALL user functions with \[]string\{\} (or "none"). Internal tools like next\_step still work.

Best practice ------------- Call SetFunctions() on EVERY step that should differ from the previous one. Treat omission as an explicit decision to inherit, not a default.

Run this file to see the rendered SWML — there are no real webhook endpoints behind the tools, this is purely a documentation example.
