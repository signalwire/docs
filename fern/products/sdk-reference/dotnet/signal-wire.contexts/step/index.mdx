  <Card title="SetEnd(bool)" href="/reference/dotnet/signal-wire.contexts/step/set-end-bool">
    Mark this step as terminal for the step flow. &lt;p>&lt;b>IMPORTANT:&lt;/b> &lt;code class="paramref">end&lt;/code> = true does NOT end the conversation or hang up the call. It exits step mode entirely after this step executes — clearing the steps list, current step index, valid\_steps, and valid\_contexts. The agent keeps running, but operates only under the base system prompt and the context-level prompt; no more step instructions are injected and no more &lt;code>next\_step&lt;/code> tool is offered.&lt;/p> &lt;p>To actually end the call, call a hangup tool or define a hangup hook.&lt;/p>

  <Card title="SetFunctions(object)" href="/reference/dotnet/signal-wire.contexts/step/set-functions-object">
    Set which non-internal functions are callable while this step is active. &lt;p>&lt;b>IMPORTANT — inheritance behavior:&lt;/b> If you do NOT call this method, the step inherits whichever function set was active on the previous step (or the previous context's last step). The server-side runtime only resets the active set when a step explicitly declares its &lt;code>functions&lt;/code> field. This is the most common source of bugs in multi-step agents: forgetting &lt;xref href="SignalWire.Contexts.Step.SetFunctions(System.Object)" data-throw-if-not-resolved="false">&lt;/xref> on a later step lets the previous step's tools leak through. Best practice is to call &lt;xref href="SignalWire.Contexts.Step.SetFunctions(System.Object)" data-throw-if-not-resolved="false">&lt;/xref> explicitly on every step that should differ from the previous one.&lt;/p> &lt;p>Keep the per-step active set small: LLM tool selection accuracy degrades noticeably past ~7-8 simultaneously-active tools per call. Use per-step whitelisting to partition large tool collections.&lt;/p> &lt;p>Internal functions (e.g. &lt;code>gather\_submit&lt;/code>, hangup hook) are ALWAYS protected and cannot be deactivated by this whitelist. The native navigation tools &lt;code>next\_step&lt;/code> and &lt;code>change\_context&lt;/code> are injected automatically when &lt;xref href="SignalWire.Contexts.Step.SetValidSteps(System.Collections.Generic.List%7bSystem.String%7d)" data-throw-if-not-resolved="false">&lt;/xref> / &lt;xref href="SignalWire.Contexts.Step.SetValidContexts(System.Collections.Generic.List%7bSystem.String%7d)" data-throw-if-not-resolved="false">&lt;/xref> is used; they are not affected by this list.&lt;/p>
