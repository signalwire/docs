## Enums

### CallState

The lifecycle state of a RELAY `Call`, as a typed, compile-time-checked
closed set.

&lt;p>The RELAY server pushes the current call state as a bare string on every
`calling.call.state` event (`created` → `ringing` →
`answered` → `ending` → `ended`).

This enum mirrors the
five values the Python reference declares in
`signalwire/relay/constants.py` as `CALL_STATE_*`
(`Constants#CALL_STATE_CREATED` … `Constants#CALL_STATE_ENDED`).

&lt;p>It is exposed &lt;em>alongside&lt;/em> the existing string getter, never instead
of it: `Call#getState()` keeps returning the raw wire string (parity
with Python, forward-compatible with any value the server later adds), while
`Call#getCallState()` returns `Optional&lt;CallState>` for callers
who want a typed, switch-exhaustive handle and a `#isTerminal()`
predicate.

&lt;pre>raw = call.getState();              // "answered" (always present, parity)
Optional&lt;CallState> typed = call.getCallState();
if (typed.map(CallState::isTerminal).orElse(false)) &#123;
    // call has reached ENDED
&#125;&lt;/pre>

&lt;p>&lt;strong>Server-emitted, growable.&lt;/strong> These values mirror what the
server sends and the set can grow in a future protocol revision. So
`#fromWire(String)` returns `null` (and `Call#getCallState()`
`Optional.empty()`) for an unrecognised value rather than throwing —
an unknown state must never crash event dispatch. This is the Java analog of
Rust's `#[non_exhaustive]` + a fallible `from_str`.

&lt;p>&lt;strong>This is a DISTINCT vocabulary from `DialState` and
`MessageState`.&lt;/strong> A `Call`'s lifecycle state
(`created/ringing/answered/ending/ended`) is not the dial &lt;em>outcome&lt;/em>
(`dialing/answered/failed`) and not a message delivery state
(`queued/…/delivered`). The three are modelled as three separate enums
and are never conflated — mirroring the three separate `*_STATE_*`
blocks in the Python reference's `constants.py`.

### DialState

The outcome state of an outbound `calling.dial`, as a typed,
compile-time-checked closed set.

&lt;p>A dial progresses `dialing` → `answered` (a leg won) or
`dialing` → `failed` (every leg gave up).

The RELAY server reports
it as the bare `dial_state` field on each `calling.call.dial`
event. This enum mirrors the three values the Java port declares in
`Constants` as `DIAL_STATE_*`
(`Constants#DIAL_STATE_DIALING` / `Constants#DIAL_STATE_ANSWERED`
/ `Constants#DIAL_STATE_FAILED`), which back the dial-completion logic
in `RelayClient.handleDialEvent`.

&lt;p>It is exposed &lt;em>alongside&lt;/em> the existing string getter, never instead
of it: `RelayEvent.CallDialEvent#getDialState()` keeps returning the raw
wire string (parity with Python, forward-compatible), while
`RelayEvent.CallDialEvent#getDialStateEnum()` returns
`Optional&lt;DialState>` for callers who want a typed handle and an
`#isTerminal()` predicate.

&lt;p>&lt;strong>Terminal semantics.&lt;/strong> Unlike `CallState` (where only
`ended` is terminal), &lt;em>both&lt;/em> `#ANSWERED` and `#FAILED`
are terminal dial outcomes — each resolves the pending dial (one completes the
`CompletableFuture&lt;Call>`, the other fails it). Only `#DIALING` is
non-terminal. So "terminal" here means "the dial has a final outcome",
matching the `answered`/`failed` branches that resolve the dial in
`RelayClient`.

&lt;pre>// "answered" (always present, parity)
event.getDialStateEnum()
     .filter(DialState::isTerminal)          // ANSWERED or FAILED
     .ifPresent(s -> ...);&lt;/pre>

&lt;p>&lt;strong>Server-emitted, growable&lt;/strong> — `#fromWire(String)`
returns `null` for an unrecognised value rather than throwing (the Java
analog of Rust's `#[non_exhaustive]` + fallible `from_str`); an
unknown dial state must never crash event dispatch.

&lt;p>&lt;strong>This is a DISTINCT vocabulary from `CallState` and
`MessageState`.&lt;/strong> A dial outcome (`dialing/answered/failed`)
is not a call lifecycle state and not a message delivery state. Note in
particular `answered` appears in &lt;em>both&lt;/em> `DialState` and
`CallState` but means different things (the dial won vs. the call leg is
up) — the two enums are never shared.

### MessageState

The delivery state of a RELAY `Message`, as a typed,
compile-time-checked closed set.

&lt;p>An outbound message progresses `queued` → `initiated` →
`sent` → `delivered` (success) or → `undelivered` /
`failed` (failure); an inbound message arrives as `received`.

The
RELAY server reports the state as the bare `message_state` field on
`messaging.state` (outbound) and `messaging.receive` (inbound)
events. This enum mirrors the seven values the Python reference declares in
`signalwire/relay/constants.py` as `MESSAGE_STATE_*`
(`Constants#MESSAGE_STATE_QUEUED` … `Constants#MESSAGE_STATE_RECEIVED`).
The `messaging.state` wire schema enumerates the six outbound values;
`received` is the inbound-only seventh.

&lt;p>It is exposed &lt;em>alongside&lt;/em> the existing string getter, never instead
of it: `Message#getState()` keeps returning the raw wire string (parity
with Python, forward-compatible), while `Message#getMessageState()`
returns `Optional&lt;MessageState>` for callers who want a typed handle and
an `#isTerminal()` predicate.

&lt;pre>raw = msg.getState();                // "delivered" (always present, parity)
msg.getMessageState()
   .filter(MessageState::isTerminal)        // DELIVERED / UNDELIVERED / FAILED
   .ifPresent(s -> ...);&lt;/pre>

&lt;p>&lt;strong>Terminal semantics.&lt;/strong> The terminal set is
`#DELIVERED`, `#UNDELIVERED`, `#FAILED` — exactly the
reference's `MESSAGE_TERMINAL_STATES` and the set
`Constants#isTerminalMessageState(String)` checks (the state at which
`Message` resolves its completion future). `received` is NOT
terminal: it is an inbound arrival, not the end of an outbound delivery
lifecycle.

&lt;p>&lt;strong>Server-emitted, growable&lt;/strong> — `#fromWire(String)`
returns `null` for an unrecognised value rather than throwing (the Java
analog of Rust's `#[non_exhaustive]` + fallible `from_str`); an
unknown message state must never crash event dispatch.

&lt;p>&lt;strong>This is a DISTINCT vocabulary from `CallState` and
`DialState`.&lt;/strong> A message delivery state is neither a call
lifecycle state nor a dial outcome; the three enums are never conflated.
