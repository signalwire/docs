# `datamap`

Fluent builder for creating SWAIG data\_map configurations.

&lt;p>
DataMap tools execute on SignalWire servers without requiring webhook endpoints.
Supports variable expansion: `$&#123;args.param`&#125;, `$&#123;response.field`&#125;,
`$&#123;global_data.key`&#125;, `$&#123;foreach.item`&#125;.
&lt;p>
All methods return `this` for fluent chaining.

### parameter

Add a parameter to this data-map tool — the `description` is
LLM-FACING.

&lt;p>Each parameter description is rendered into the OpenAI tool schema
under `parameters.properties.&lt;name>.description` and sent to
the model.

The model uses it to decide HOW to fill in the argument
from user speech. It is prompt engineering, not developer FYI.

&lt;h3>Bad vs good&lt;/h3>
&lt;pre>: .parameter("city", "string", "the city", true, null)
GOOD: .parameter("city", "string",
          "The name of the city to get weather for, e.g. "
          + "\"San Francisco\". Ask the user if they did not "
          + "provide one. Include the state or country if the "
          + "city name is ambiguous.", true, null)&lt;/pre>

### purpose

Set the LLM-facing tool description — this is PROMPT ENGINEERING,
not developer documentation.

&lt;p>The description string is rendered into the OpenAI tool schema
`description` field on every LLM turn.

The model reads it to
decide WHEN to call this tool. A vague `purpose()` is the #1
cause of "the model has the right tool but doesn't call it" failures
with data-map tools.

&lt;h3>Bad vs good&lt;/h3>
&lt;pre>: .purpose("weather api")
GOOD: .purpose("Get the current weather conditions and forecast "
             + "for a specific city. Use this whenever the user "
             + "asks about weather, temperature, rain, or similar "
             + "conditions in a named location.")&lt;/pre>

#### Signature

```java
public DataMap purpose(String description)
```
