# `ContextBuilder`

Builder for multi-step, multi-context AI agent workflows.

&lt;p>A ContextBuilder owns one or more `Context`s; each context owns an
ordered list of `Step`s.

Only one context and one step is active at a
time. Per chat turn, the runtime injects the current step's instructions as
a system message, then asks the LLM for a response.

&lt;h2>Native tools auto-injected by the runtime&lt;/h2>

&lt;p>When a step (or its enclosing context) declares valid_steps or
valid_contexts, the runtime auto-injects two native tools so the model can
navigate the flow:
&lt;ul>
  &lt;li>enum)        ŌĆö present when valid_steps is set&lt;/li>
  &lt;li>enum) ŌĆö present when valid_contexts is set&lt;/li>
&lt;/ul>

&lt;p>A third native tool ŌĆö `gather_submit` ŌĆö is injected during
gather_info questioning. These three names are &lt;b>reserved&lt;/b>:
`#validate()` rejects any agent that defines a SWAIG tool with one of
these names. See `#RESERVED_NATIVE_TOOL_NAMES`.

&lt;h2>Function whitelisting (Step.setFunctions)&lt;/h2>

&lt;p>Each step may declare a functions whitelist. The whitelist is applied
in-memory at the start of each LLM turn. CRITICALLY: if a step does NOT
declare a functions field, it INHERITS the previous step's active set.
See `Step#setFunctions(Object)` for details and examples.
