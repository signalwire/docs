SWML provides a powerful variable system that allows you to access call information, store data, and pass parameters between sections.
This page is the authoritative technical reference for global variable scopes (`call`, `params`, `vars`, `envs`) and variable management in SWML.

For information about using JavaScript expressions with variables, see the [Expressions Reference](/docs/swml/reference/expressions).
For template transformation functions, see the [Template Functions Reference](/docs/swml/reference/template-functions).

## Syntax

SWML uses the `${variable}` syntax for variable substitution:

<CodeBlocks>
<CodeBlock title="YAML">
```yaml
version: 1.0.0
sections:
  main:
    - play:
        url: 'say: This call is from the number ${call.from}'
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": {
    "main": [
      {
        "play": {
          "url": "say: This call is from the number ${call.from}"
        }
      }
    ]
  }
}
```
</CodeBlock>
</CodeBlocks>

## Reference

SWML provides several variable scopes that are available throughout your scripts. These scopes give you access to call information, script parameters, and stored data.

<ParamField path="call" type="object" toc={true}>
  Information about the current call. Contains the following properties. 
  
  **Scope:** Call-specific and read-only. Each call leg (A-leg, B-leg) has its own unique `call` object with different `call_id`, `from`, `to`, etc.
   When connecting to a new leg, the `call` object is re-initialized with the new leg's data.
</ParamField>

<ParamField path="envs" type="object" toc={true}>
  Environment variables configured at the account or project level in your SignalWire configuration. These provide access to account-wide settings and configuration values.
  
  <Warning>
Coming soon!
The `envs` object is included in POST request bodies to external servers, but the ability to set environment variables in the SignalWire Dashboard is not yet available in production. This feature is coming soon.
</Warning>
</ParamField>

  **Scope:** Account/project-level and read-only. Set in your SignalWire account configuration, not within SWML scripts.

  **Fallback behavior:** When you reference a variable without a scope prefix (e.g., `${my_variable}`), SWML first checks `vars`. If not found in `vars`, it automatically falls back to `envs`.

  **Example keys:** `envs.api_key`, `envs.webhook_url`, `envs.account_setting`

<ParamField path="params" type="object" toc={true}>
  Parameters that are user-defined and set by the calling the [`execute`](/docs/swml/reference/calling/execute) or [`transfer`](/docs/swml/reference/calling/transfer) method.
</ParamField>

  **Scope:** Section-scoped and read-only. Each section has its own independent `params` that must be explicitly passed via `execute` or `transfer`. Params do not persist after a section completes.
  When an `execute` call returns, the calling section's original params are restored.

  **Example keys:** `params.audio_file`, `params.volume`, `params.department`

<ParamField path="vars" type="object" toc={true}>
  Script variables that can be set, modified, and accessed throughout your SWML script.
</ParamField>

  **Created by:**
  - [`set`](/docs/swml/reference/calling/set) method - Create or update user-defined variables
  - Method outputs - Many methods automatically create variables (e.g., `prompt_value`, `record_url`, `return_value`)

  **Managed with:**
  - [`unset`](/docs/swml/reference/calling/unset) method - Remove variables

  **Example keys:** `vars.user_choice`, `vars.counter`, `vars.prompt_value`, `vars.record_url`

  **Scope:** Global within a single call session. Variables persist across all sections and through `execute` calls.
  However, connecting to a new leg of a call will reset the `vars` object to an empty state.

  **Variable access:** Variables can be accessed with or without the `vars.` prefix.
  When you reference a variable without a scope prefix (e.g., `${my_variable}`), SWML first checks `vars`. If not found in `vars`, it automatically falls back to `envs`.

</Tab>
<Tab title="Messaging">

Variables exposed to a messaging SWML document handling an inbound message.
Messaging SWML uses `%{variable.path}` for variable substitution (rather than the `${variable}` syntax used by calling SWML).

<ParamField path="message" type="object" toc={true}>
  Information about the inbound message that triggered the SWML document, exposed to `%{...}` variable
  expansion. Each inbound message produces its own `message` object.

  **Scope:** Message-specific and read-only. There is no persistent state between inbound messages.

  <Note>
    The `message` object available to `%{...}` variable expansion is a **subset** of the inbound
    message webhook payload your server receives. The expansion context exposes the six fields below.
    For the full payload shape (including `message_id`, `space_id`, `direction`, `media[]`,
    `segments`, and `timestamp`), see the
    [SWML inbound message webhook](/docs/apis/rest/swml-webhook/webhooks/inbound-message-webhook).
  </Note>
</ParamField>

  <Indent>

    <ParamField path="message.to" type="string" toc={true}>
      Recipient phone number in E.164 format (the SignalWire number that received the message).
    </ParamField>

    <ParamField path="message.from" type="string" toc={true}>
      Sender phone number in E.164 format.
    </ParamField>

    <ParamField path="message.body" type="string" toc={true}>
      The text content of the inbound message.
    </ParamField>

    <ParamField path="message.project_id" type="string" toc={true}>
      The Project ID this message belongs to.
    </ParamField>

    <ParamField path="message.kind" type="string" toc={true}>
      The message kind. Possible values: `sms`, `mms`.
    </ParamField>

    <ParamField path="message.status" type="string" toc={true}>
      The inbound message status. At the start of SWML execution this is `received`; the value is
      captured once when the document begins executing, so subsequent state transitions are not
      reflected in this variable.
    </ParamField>

  </Indent>

<ParamField path="params" type="object" toc={true}>
  Parameters passed via the [`execute`](/docs/swml/reference/messaging/execute) or [`transfer`](/docs/swml/reference/messaging/transfer) method.
</ParamField>

  **Scope:** Section-scoped and read-only. Each `execute` or `transfer` call replaces (does not merge with) the caller's `params`. When the called section returns, the caller's original `params` are restored.

  **Example keys:** `params.reason`, `params.source`, `params.body`

<ParamField path="reply_result" type="string" toc={true}>
  Result of the most recent [`reply`](/docs/swml/reference/messaging/reply) step. Possible values: `queued`, `failed`.
</ParamField>

<ParamField path="reply_message_id" type="string" toc={true}>
  Message segment ID of the most recent successful `reply`.
</ParamField>

<ParamField path="request_result" type="string" toc={true}>
  Result of the most recent [`request`](/docs/swml/reference/messaging/request) step. Possible values: `success`, `failed`, `timeout`, `limit_exceeded`.
</ParamField>

<ParamField path="request_response" type="object" toc={true}>
  Parsed JSON response from the most recent `request`, populated only when `save_variables: true` was set on the `request` step.
  Access individual fields with dot notation, e.g. `%{request_response.field_name}`.
</ParamField>

<ParamField path="request_response_code" type="integer" toc={true}>
  HTTP response code from the most recent `request`.
</ParamField>

<ParamField path="request_response_body" type="string" toc={true}>
  Raw response body from the most recent `request`. Truncated to 64 KB.
</ParamField>

<ParamField path="return_value" type="any" toc={true}>
  Value passed by the most recent [`return`](/docs/swml/reference/messaging/return) from a section called via `execute`.
</ParamField>

</Tab>

## Variables in serverless and server-based SWML

All SWML variables (`call`, `params`, `vars`, `envs`) are available in both serverless (Dashboard-hosted) and server-based (external URL) deployments.

### Serverless (dashboard-hosted) scripts

When SWML is executed directly from the SignalWire Dashboard, access variables using the `${}` syntax:

<CodeBlocks>
<CodeBlock title="YAML">
```yaml
version: 1.0.0
sections:
  main:
    - set:
        department: sales
    - play:
        url: 'say: You are calling from ${call.from}'
    - play:
        url: 'say: Department is ${vars.department}'
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": {
    "main": [
      {
        "set": {
          "department": "sales"
        }
      },
      {
        "play": {
          "url": "say: You are calling from ${call.from}"
        }
      },
      {
        "play": {
          "url": "say: Department is ${vars.department}"
        }
      }
    ]
  }
}
```
</CodeBlock>
</CodeBlocks>

### Server-based (external URL) scripts

When SWML is served from your web server, SignalWire sends the current variable state as JSON in the POST request body (see the [JSON format example](#example-variables-in-json-format) above).

You have two options for working with these variables in your SWML response:

#### Option 1: Use variable expansion syntax

Use `${}` syntax in your returned SWML, and SignalWire will substitute the values at runtime:

<CodeBlocks>
<CodeBlock title="YAML">
```yaml
version: 1.0.0
sections:
  main:
    - play:
        url: 'say: Welcome to ${params.department}'
    - play:
        url: 'say: Calling from ${call.from}'
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": {
    "main": [
      {
        "play": {
          "url": "say: Welcome to ${params.department}"
        }
      },
      {
        "play": {
          "url": "say: Calling from ${call.from}"
        }
      }
    ]
  }
}
```
</CodeBlock>
</CodeBlocks>

#### Option 2: Extract and insert values server-side

Extract variables from the request body in your server code and insert them directly into the SWML response:

```javascript
// Example: Node.js/Express server
app.post('/swml-handler', (req, res) => {
  const { call, vars, envs, params } = req.body;

  // Extract values from the request
  const department = params.department || 'support';
  const callerNumber = call.from;
  const apiKey = envs.api_key;

  // Build SWML with values inserted directly
  const swml = {
    version: '1.0.0',
    sections: {
      main: [
        {
          play: {
            url: `say: Welcome to ${department}`
          }
        },
        {
          play: {
            url: `say: Calling from ${callerNumber}`
          }
        }
      ]
    }
  };

  res.json(swml);
});
```

Both approaches produce the same result. Use variable expansion (`${}`) for simpler cases, or extract values server-side when you need to perform logic or transformations on the data.

See the [Deployment Guide](/docs/swml/guides/deployment) for complete server setup instructions.

## Accessing variable data

Variables can contain different types of data - simple values, nested objects, or arrays. Use dot notation (`.`) for object properties, bracket notation (`[]`) for array elements, or combine both for complex data structures.

### Simple values

Access variables directly by name:

<CodeBlocks>
<CodeBlock title="YAML">
```yaml
version: 1.0.0
sections:
  main:
    - set:
        name: Alice
        age: 30
    - play:
        url: 'say: Hello ${name}, you are ${age} years old'
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": {
    "main": [
      {
        "set": {
          "name": "Alice",
          "age": 30
        }
      },
      {
        "play": {
          "url": "say: Hello ${name}, you are ${age} years old"
        }
      }
    ]
  }
}
```
</CodeBlock>
</CodeBlocks>

### Nested objects

Access nested properties using dot notation:

<CodeBlocks>
<CodeBlock title="YAML">
```yaml
version: 1.0.0
sections:
  main:
    - set:
        user:
          name: Alice
          address:
            city: Seattle
            state: WA
    - play:
        url: 'say: ${user.name} lives in ${user.address.city}, ${user.address.state}'
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": {
    "main": [
      {
        "set": {
          "user": {
            "name": "Alice",
            "address": {
              "city": "Seattle",
              "state": "WA"
            }
          }
        }
      },
      {
        "play": {
          "url": "say: ${user.name} lives in ${user.address.city}, ${user.address.state}"
        }
      }
    ]
  }
}
```
</CodeBlock>
</CodeBlocks>

### Arrays

Access array elements using bracket notation with zero-based indexing:

<CodeBlocks>
<CodeBlock title="YAML">
```yaml
version: 1.0.0
sections:
  main:
    - set:
        departments:
          - Sales
          - Support
          - Engineering
    - play:
        url: 'say: First department is ${departments[0]}'
    - play:
        url: 'say: Second department is ${departments[1]}'
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": {
    "main": [
      {
        "set": {
          "departments": [
            "Sales",
            "Support",
            "Engineering"
          ]
        }
      },
      {
        "play": {
          "url": "say: First department is ${departments[0]}"
        }
      },
      {
        "play": {
          "url": "say: Second department is ${departments[1]}"
        }
      }
    ]
  }
}
```
</CodeBlock>
</CodeBlocks>

### Arrays of objects

Combine bracket and dot notation to access properties in array elements:

<CodeBlocks>
<CodeBlock title="YAML">

```yaml
sections:
  main:

          - name: Alice
            role: Engineer

            role: Manager
    - play:
        url: 'say: ${employees[0].name} is an ${employees[0].role}'
    - play:
        url: 'say: ${employees[1].name} is a ${employees[1].role}'
```
</CodeBlock>
