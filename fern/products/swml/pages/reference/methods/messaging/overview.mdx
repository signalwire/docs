Messaging SWML is the flavor of SWML used to handle **inbound SMS and MMS messages**. When a
message arrives on a phone number configured with a SWML message handler, SignalWire fetches and
processes the SWML document, optionally sending an outbound reply.

For handling voice calls, see the [Calling SWML overview](/docs/swml/reference/calling).

## Document modes

Messaging SWML supports two document shapes:

- **Light mode** — a single top-level [`reply`](/docs/swml/reference/messaging/reply) or
  [`receive`](/docs/swml/reference/messaging/receive) key, for one-shot handling. An empty
  document (`{}`) is treated as an implicit `receive`.
- **Full mode** — a top-level `sections` map with named sections, for multi-step flows with
  branching, subroutines, or retries. Execution starts at `sections.main`, and each section
  contains an array of methods that execute sequentially. A document can execute up to **100
  method steps** per inbound message; once that ceiling is hit, execution stops.

### Full mode properties

<ParamField path="version" type="string" default="1.0.0" toc={true}>
  The SWML schema version. Currently only `"1.0.0"` is accepted.
</ParamField>

<ParamField path="sections" type="object" required={true} toc={true}>
  A map of named sections. Each value is an array of [methods](#methods) that run sequentially.
  Must contain at least a `main` section; additional named sections can be invoked via
  [`execute`](/docs/swml/reference/messaging/execute) or
  [`goto`](/docs/swml/reference/messaging/goto).
</ParamField>

<Indent>

<ParamField path="sections.main" type="array" required={true} toc={true}>
  The entry-point section. Execution starts here when the document loads. Each item in the
  array is one of the [methods](#methods) listed below.
</ParamField>


<ParamField path="sections.<name>" type="array" toc={true}>
  Any additional named section, invokable via `execute` or `goto`. Each item in the array is
  one of the [methods](#methods) listed below.
</ParamField>

</Indent>

### Light mode properties

A light-mode document contains **exactly one** top-level key — either `reply` or `receive` —
or the empty document `{}`. Light-mode documents do not have a `version` or `sections` wrapper.

<ParamField path="reply" type="string | object" toc={true}>
  Send an outbound message in response to the inbound message and stop. Accepts a string
  shorthand (the reply body), an object with body/media/routing fields, or an inline `switch`
  that branches on a variable. See [`reply`](/docs/swml/reference/messaging/reply) for the full
  field reference.
</ParamField>

<ParamField path="receive" type="object" toc={true}>
  Accept the inbound message without sending a reply. Pass an empty object (`{ "receive": {} }`),
  or omit the key entirely by sending an empty document (`{}`) — both are equivalent. See
  [`receive`](/docs/swml/reference/messaging/receive).
</ParamField>

When SignalWire fetches a Messaging SWML document from an external URL, it sends a POST request
with the inbound message webhook payload.
This applies on the initial inbound-message fetch and on every fetch triggered by a
[`transfer`](/docs/swml/reference/messaging/transfer) step.

    the REST API reference rebuilds. */}

```json
{
  "message": {
    "message_id": "c2d3e4f5-a6b7-8901-cdef-234567890abc",
    "project_id": "b2c3d4e5-f6a7-8901-bcde-f12345678901",
    "space_id": "d3e4f5a6-b7c8-9012-defa-345678901bcd",
    "direction": "inbound",
    "type": "sms",
    "from": "+15551231234",
    "to": "+15553214321",
    "body": "Hello, I need help",
    "media": [],
    "segments": 1,
    "timestamp": "2024-01-15T10:30:00Z"
  },
  "params": {}
}
```

The payload includes:

- **`message`** — the inbound message that triggered the fetch (`from`, `to`, `body`, `media`,
  `type`, `segments`, `timestamp`, etc.).
- **`params`** — parameters supplied by the calling
  [`transfer`](/docs/swml/reference/messaging/transfer) step. Empty `{}` on the initial inbound fetch.
- **`vars`** — runtime variables propagated from the originating document
  (`request_result`, `reply_result`, `request_response`, `request_response_code`,
  `request_response_body`, `reply_message_id`). Present only on `transfer`-driven fetches;
  **absent** on the initial inbound fetch.

Your server must respond with a valid SWML document using one of these content types:
`application/json`, `application/yaml`, or `text/x-yaml`.

## Methods

A method is a single step in a full-mode SWML document — each item in a `sections.<name>` array
invokes one method. Each method's reference page documents its parameters, defaults, and any
output variables it sets. Methods below are grouped by what they do.

### Message handling

Decide what to do with the inbound message — send an outbound reply, or accept it silently.
Both methods can also appear at the top level of a [light-mode document](#light-mode-properties).

<CardGroup cols={2}>
  <Card title="reply" href="/docs/swml/reference/messaging/reply">
    Send an SMS or MMS back to the sender of the inbound message. Use for auto-replies, confirmations, and keyword responses.
  </Card>
  <Card title="receive" href="/docs/swml/reference/messaging/receive">
    Accept the inbound message without sending a reply. Use to acknowledge the message without an outbound response.
  </Card>
</CardGroup>

### Control flow

Direct execution within the document — reuse blocks of logic, repeat steps, choose between
branches, or hand off the message to another document entirely.

<CardGroup cols={2}>
  <Card title="execute" href="/docs/swml/reference/messaging/execute">
    Run another section of the current document, then resume when it finishes. Use to reuse a sequence of steps from more than one place.
  </Card>
  <Card title="return" href="/docs/swml/reference/messaging/return">
    Hand control back to the section that called the current one, optionally passing a value the caller can read.
  </Card>
  <Card title="transfer" href="/docs/swml/reference/messaging/transfer">
    Hand off the rest of the conversation to a different SWML document hosted on an external server. The current document stops; the new one takes over.
  </Card>
  <Card title="goto" href="/docs/swml/reference/messaging/goto">
    Restart from a named point earlier or later in the current section. Use for retry loops that repeat a step until it succeeds.
  </Card>
  <Card title="label" href="/docs/swml/reference/messaging/label">
    Mark a place in the section that `goto` can jump to. Labels perform no action; they serve as navigation anchors.
  </Card>
  <Card title="switch" href="/docs/swml/reference/messaging/switch">
    Choose what to do next based on a value from the inbound message or a variable. Use for keyword routing — different replies for different inbound message bodies.
  </Card>
</CardGroup>

### External integration

Reach out to your own backend during message handling, so the reply can depend on data only your
systems know.

<CardGroup cols={2}>
  <Card title="request" href="/docs/swml/reference/messaging/request">
    Call an external API while handling the message — for example, to look up a customer record before replying.
  </Card>
</CardGroup>

## Variable expansion

SWML supports `%{variable.path}` placeholders in string values. Variables are expanded after the
document is parsed, before methods execute. Expansion is **pure substitution only** — no expressions,
method calls, or operators. Unresolvable paths are replaced with an empty string.

The inbound `message.*` context (`message.to`, `message.from`, `message.body`, `message.kind`,
`message.project_id`, `message.status`) is available throughout the document. See
[Variables](/docs/swml/reference/variables) for the full variable reference.
