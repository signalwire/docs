Create and send an outbound message in response to the inbound message. Accepts either a string shorthand
(used as the message body) or an object with body, optional media, and optional routing fields.

`reply` does **not** end execution — subsequent steps in the section continue to run after the reply
is queued.

## **Properties**

`reply` accepts either a string (used as the message body) or an object with the following properties.

<ParamField path="reply" type="string | object" required={true} toc={true}>
  String shorthand or an object containing the properties below. At least one of `body` or `media` must
  be present when using the object form.
</ParamField>

<Indent>

<ParamField path="reply.body" type="string" toc={true}>
  The body of the reply message. Required if `media` is not present.
</ParamField>


<ParamField path="reply.media" type="string[]" toc={true}>
  Array of media URLs to attach. Converts the message to MMS. Maximum 8 attachments. Required if `body`
  is not present.
</ParamField>


<ParamField path="reply.to" type="string" toc={true}>
  Destination phone number in E.164 format. Defaults to the inbound message's `from`.
</ParamField>


<ParamField path="reply.from" type="string" toc={true}>
  Sending phone number — must be a PhoneRoute or ShortCode in the project. Defaults to the inbound
  message's `to`.
</ParamField>


<ParamField path="reply.status_url" type="string" toc={true}>
  URL to receive delivery status callbacks for the outbound reply message. See
  [Status callbacks](#status-callbacks) below.
</ParamField>

</Indent>

## **Variables**

Set by the method:

- **reply_result:** (out) `queued` | `failed`.
- **reply_message_id:** (out) Message segment ID of the last successful reply.

If `reply` is called multiple times in a single document, these variables reflect the most recent
reply.

## **Status callbacks**

The `status_url` on `reply` registers a delivery status callback for the **outbound reply message** —
not for the inbound message that triggered the SWML document. It behaves the same way as the status
callback on any other outbound message sent through SignalWire.

- Fires as the outbound reply moves through delivery states.
- Callback delivery is independent of SWML execution: the SWML document completes as soon as the reply
  is accepted (`reply_result: "queued"`); delivery-state callbacks fire afterwards.
- If `reply` is called multiple times in a single document, each reply's `status_url` is registered
  independently for its own outbound message.

When `status_url` is set, SignalWire sends an HTTP `POST` to that URL each time the outbound reply
transitions to a new state. The callback uses the same payload as other outbound messages sent
through SignalWire:

<WebhookPayloadSnippet webhook="messageStatusCallback" />

See the [Message status callback](/docs/apis/rest/messages/webhooks/message-status-callback)
webhook page for the full field reference and the list of possible `status` values.

## **Examples**

### Simple-mode string shorthand

<CodeBlocks>
<CodeBlock title="YAML">
```yaml
reply: "Thanks for your message!"
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "reply": "Thanks for your message!"
}
```
</CodeBlock>
</CodeBlocks>

### Reply with body

<CodeBlocks>
<CodeBlock title="YAML">
```yaml
version: 1.0.0
sections:
  main:
    - reply:
        body: "Thanks for your message!"
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": {
    "main": [
      {
        "reply": {
          "body": "Thanks for your message!"
        }
      }
    ]
  }
}
```
</CodeBlock>
</CodeBlocks>

### Reply with media (MMS)

<CodeBlocks>
<CodeBlock title="YAML">
```yaml
version: 1.0.0

  main:
    - reply:
        body: "Here's the document you requested"
        media:
          - "https://example.com/document.pdf"
```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": {
    "main": [
      {
        "reply": {
          "body": "Here's the document you requested",
          "media": [
            "https://example.com/document.pdf"
          ]
        }
      }
    ]
  }
}
```
</CodeBlock>
</CodeBlocks>

### Forward an inbound message to another number

<CodeBlock title="YAML">

```yaml
  main:
    - reply:

```
</CodeBlock>
<CodeBlock title="JSON">
```json
{
  "version": "1.0.0",
  "sections": {
    "main": [
      {
        "reply": {
          "to": "+12223334444",
          "from": "+15559876543",
          "body": "Your number %{message.to} got a message from %{message.from}! The body was: %{message.body}"
        }
      }
    ]
  }
}
```
</CodeBlock>
</CodeBlocks>

## **Failure behavior**

`reply` validation failures (invalid `from`, invalid `to`, missing body/media, exceeding the character
limit, inactive campaign, etc.) mark the inbound message as `failed` with a specific error code and do
**not** create a partial outbound message. Common error codes include
`swml_reply_invalid_from_number`, `swml_reply_invalid_to_number`,
`swml_reply_body_or_media_required`, `swml_reply_media_limit_exceeded`,
`swml_reply_character_limit_exceeded`, `swml_reply_missing_messaging_capability`, and
`swml_reply_inactive_campaign`.
