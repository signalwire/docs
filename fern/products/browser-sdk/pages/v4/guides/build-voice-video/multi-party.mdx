A "room" in SignalWire is a call with more than two members. The
`Call` API is the same тАФ `dial()` into a room address, get back a
`Call`, observe `remoteStream$` for the mixed feed тАФ but several
observables that don't matter in a 1:1 call become central:

- `call.participants$` тАФ everyone in the room, including `self`.
- `Participant.isTalking$` тАФ voice activity per member.
- `Participant.handraised$` тАФ hand-raise state per member.
- `Participant.position$` / `call.layoutLayers$` тАФ where each member
  is rendered (see [Layouts](/docs/browser-sdk/v4/guides/layouts)).

Moderation actions (mute someone else, remove them, lock the room)
sit on [`Participant`] and `Call`, gated by capabilities.

## Joining

A room is just a Resource Address with `type === 'room'`. Dial it the
same way as any other call:

```js
const call = await client.dial("/public/team-standup", {
  audio: true,
  video: true,
  receiveAudio: true,
  receiveVideo: true,
});
```

The mixed video feed comes through `remoteStream$` тАФ bind it to a
single `<video>` element. The composition (grid, presenter, etc.) is
controlled server-side; see [Layouts](/docs/browser-sdk/v4/guides/layouts).

## The participants list

`participants$` is a BehaviorSubject of the current member list. It
re-emits on join, leave, and updates:

```js
const tiles = new Map();

call.participants$.subscribe((participants) => {
  // Remove tiles for members who left
  for (const [id, el] of tiles) {
    if (!participants.find((p) => p.id === id)) {
      el.remove();
      tiles.delete(id);
    }
  }

  // Add tiles for new members
  for (const p of participants) {
    if (!tiles.has(p.id)) {
      const el = renderTile(p);
      tiles.set(p.id, el);
      participantsList.appendChild(el);
      bindParticipantObservables(p, el);
    }
  }
});
```

The `participants$` list includes the self participant. Filter on
`p.id === call.self?.id` if you want to treat self specially in your
UI.

## Per-participant observables

Each [`Participant`] exposes its own reactive state. The full set:

| Observable                 | What it carries                                  |
| -------------------------- | ------------------------------------------------ |
| `name$`                    | Display name                                     |
| `type$`                    | Member type string (e.g. `'member'`, `'audience'`)|
| `audioMuted$`              | Whether their audio is muted                     |
| `videoMuted$`              | Whether their video is muted                     |
| `deaf$`                    | Whether they've muted incoming audio for themselves |
| `handraised$`              | Hand raise state                                 |
| `isTalking$`               | Voice activity (server-side VAD)                 |
| `visible$`                 | Visible in the current layout                    |
| `position$`                | Current layout layer (x, y, width, height, тАж)    |
| `inputVolume$` / `outputVolume$` / `inputSensitivity$` | Server-side audio levels |
| `echoCancellation$` / `autoGain$` / `noiseSuppression$` | Audio processing flags |
| `userId$` / `addressId$`   | Their User and Address IDs (if any)              |
| `meta$`                    | Custom metadata (room-specific)                  |

Bind them like any RxJS observable:

```js
function bindParticipantObservables(p, el) {
  p.name$.subscribe((name) => (el.querySelector(".name").textContent = name ?? ""));
  p.audioMuted$.subscribe((m) => el.classList.toggle("muted", m));
  p.videoMuted$.subscribe((m) => el.classList.toggle("video-off", m));
  p.isTalking$.subscribe((t) => el.classList.toggle("speaking", t));
  p.handraised$.subscribe((h) => el.classList.toggle("hand-up", h));
}
```

## Moderation

Mute, remove, and reposition another participant by calling methods
on their [`Participant`] instance:

```js
// Mute their mic (they see themselves muted)
await participant.mute();
await participant.unmute();

// Remove them from the call
await participant.remove();

// Move them to a layout slot
await participant.setPosition("reserved-0");

// Server-side volume / sensitivity on their leg
await participant.setAudioInputVolume(50);
await participant.setAudioInputSensitivity(20);
```

**Always check capabilities first.** `call.self.capabilities.member`
tells you which actions the server has granted you against other
members:

```js
call.self?.capabilities.member$.subscribe((member) => {
  kickButton.disabled = !member.remove;
  muteOthersButton.disabled = !member.muteAudio.on;
  setPositionMenu.hidden = !member.position;
});
```

Trying without permission throws server-side. See
[Capabilities](/docs/browser-sdk/v4/guides/capabilities).

## End the call for everyone

`self.end()` ends the call session for *every* participant, not just
self. Gated by `call.self.capabilities.end`:

```js
endRoomButton.onclick = async () => {
  if (!confirm("End this call for everyone?")) return;
  await call.self.end();
};

call.self?.capabilities.end$.subscribe((canEnd) => {
  endRoomButton.hidden = !canEnd;
});
```

To leave the call without ending it, use `self.remove()` or
`call.hangup()`.

## Lock / unlock

A locked room rejects new joiners:

```js
await call.toggleLock();
call.locked$.subscribe((locked) => {
  lockIndicator.textContent = locked ? "ЁЯФТ Locked" : "Open";
});
```

Useful for "we've started, no more arrivals" workflows. Capability:
`call.self.capabilities.lock`.

## Lobby / waiting room patterns

The platform doesn't ship a hard "lobby" primitive in v4 тАФ implement
it as two Resources:

1. A **lobby Resource** that all guests dial first. Configure it with
   a SWML script that holds them (e.g. hold music + a recorded
   message).
2. The **main room** Resource. A moderator's app, watching
   `participants$` on the lobby, transfers individuals into the main
   room when ready:
   ```js
   await lobbyCall.transfer({ destination: "/private/main-room" });
   ```

Alternatively, lock the main room and unlock it manually when guests
arrive, accepting that everyone hears "locked" feedback until you
flip it.

## Talking heads from `isTalking$`

A common UI: highlight whoever is speaking and float them to the
front. The signal is per-participant `isTalking$`:

```js
const isTalkingMap = new Map();

call.participants$.subscribe((participants) => {
  for (const p of participants) {
    p.isTalking$.subscribe((talking) => {
      isTalkingMap.set(p.id, talking);
      const activeSpeaker = [...isTalkingMap.entries()]
        .find(([, talking]) => talking)?.[0];
      highlightSpeaker(activeSpeaker);
    });
  }
});
```

Combine with `layoutLayers$` if you want to draw an overlay border
on the active speaker's tile within the mixed video.

## A complete participant tile

The kitchen-sink demo's `subscribeToParticipantObservables` (in
`playground/kitchen-sink-demo/src/main.ts`) wires every participant
observable to a card template. Cribbing from that:

```js
function renderParticipantCard(p) {
  const card = document.createElement("div");
  card.className = "participant-card";

  // Static structure
  card.innerHTML = `
    <div class="avatar"></div>
    <div class="name"></div>
    <div class="badges">
      <span class="badge audio"></span>
      <span class="badge video"></span>
      <span class="badge speaking" hidden>ЁЯЧг</span>
      <span class="badge hand"     hidden>тЬЛ</span>
    </div>
  `;

  // Reactive bindings
  p.name$.subscribe((n)        => card.querySelector(".name").textContent = n ?? "Unknown");
  p.audioMuted$.subscribe((m)  => card.querySelector(".audio").textContent = m ? "ЁЯФЗ" : "ЁЯОЩ");
  p.videoMuted$.subscribe((m)  => card.querySelector(".video").textContent = m ? "ЁЯУ╡" : "ЁЯУ╣");
  p.isTalking$.subscribe((t)   => card.querySelector(".speaking").hidden = !t);
  p.handraised$.subscribe((h)  => card.querySelector(".hand").hidden = !h);

  return card;
}
```

Re-render the whole list on `participants$`, but bind individual
observables once per card тАФ they handle their own re-emissions.
