    functions: typing.Optional[typing.Union[str, typing.List[str]]] = None,

) -> signalwire.core.contexts.Step

<Indent>

Add a new step to this context.

When called with only ``name`` the returned Step can be configured
with the usual method-chaining API.  When the optional keyword
arguments are supplied the step is fully configured in one call:

**Parameters:**

<ParamField path="name" type="str">
Step name (must be unique within the context).
</ParamField>

<ParamField path="task" type="Optional[str]" default="None">
Text for the "Task" section (equivalent to
``step.add_section("Task", task)``).
</ParamField>

<ParamField path="bullets" type="Optional[List[str]]" default="None">
List of bullet strings for the "Process" section
(equivalent to ``step.add_bullets("Process", bullets)``).
Requires *task* to also be set.
</ParamField>

<ParamField path="criteria" type="Optional[str]" default="None">
Step-completion criteria (equivalent to
``step.set_step_criteria(criteria)``).
</ParamField>

<ParamField path="functions" type="Optional[Union[str, List[str]]]" default="None">
Tool names the step may call, or ``"none"``
(equivalent to ``step.set_functions(functions)``).
</ParamField>

<ParamField path="valid_steps" type="Optional[List[str]]" default="None">
Names of steps the agent may transition to
(equivalent to ``step.set_valid_steps(valid_steps)``).
</ParamField>

**Returns:** `Step`

The configured Step object for optional further chaining.


</Indent>

<Anchor id="signalwire-core-contexts-Step-set_gather_info">

<CodeBlock links={{"signalwire.core.contexts.Step":"#signalwire-core-contexts-Step"}} showLineNumbers={false} wordWrap={true}>

```python
signalwire.core.contexts.Step.set_gather_info(
    output_key: typing.Optional[str] = None,
    completion_action: typing.Optional[str] = None,
    prompt: typing.Optional[str] = None
) -> signalwire.core.contexts.Step
```

</CodeBlock>
</Anchor>

<Indent>

Enable info gathering for this step. Questions are presented one at a time
via dynamic step instruction re-injection, producing zero tool_call/tool_result
entries in LLM-visible history.

After calling this, use add_gather_question() to define questions.

**Parameters:**

<ParamField path="output_key" type="Optional[str]" default="None">
Key in global_data to store answers under (default: top-level)
</ParamField>

<ParamField path="completion_action" type="Optional[str]" default="None">
Where to go when all questions are answered. Can be:
- "next_step" to auto-advance to the next sequential step
- A specific step name (e.g. "process_results") to jump to that step
- None (default) to return to normal step mode after gathering
The target must be valid: "next_step" requires a following step,
and named steps must exist in the same context.
</ParamField>

<ParamField path="prompt" type="Optional[str]" default="None">
Preamble text injected once when entering the gather step, giving
the model personality/context for why it is asking these questions
</ParamField>

**Returns:** `Step`

Self for method chaining


</Indent>
